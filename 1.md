## 面试辅导计划

### 一、Java 基础高频面试题（必须掌握）
1. HashMap 底层原理（数组+链表+红黑树，扩容机制，线程安全问题）
2. ConcurrentHashMap 原理（JDK7 分段锁 vs JDK8 CAS+synchronized）
3. JVM 内存模型（堆、栈、方法区、GC 算法、GC 调优）
4. Java 多线程（线程池参数、synchronized vs ReentrantLock、volatile、AQS）
5. Java IO/NIO/Netty 基础

### 二、Spring 全家桶（重点）
1. **Spring IOC / DI** - Bean 生命周期、作用域、循环依赖解决
2. **Spring AOP** - 动态代理（JDK vs CGLIB）、切面应用场景
3. **Spring Boot** - 自动配置原理、starter 机制、常用注解
4. **Spring Cloud 核心组件：**
   - **服务注册与发现**：Eureka / Nacos
   - **负载均衡**：Ribbon / Spring Cloud LoadBalancer
   - **远程调用**：Feign / OpenFeign
   - **熔断限流**：Hystrix / Sentinel
   - **网关**：Spring Cloud Gateway
   - **配置中心**：Nacos Config / Spring Cloud Config
   - **分布式事务**：Seata

### 三、消息队列（Kafka & RocketMQ）
1. **Kafka 核心概念：**
   - Topic / Partition / Consumer Group / Offset
   - 高吞吐原理：顺序写磁盘、零拷贝、批量发送
   - 消息可靠性：acks 参数、ISR 机制、消费者提交策略
   - 消息有序性：分区有序 vs 全局有序
   - 重复消费 & 幂等性处理

2. **RocketMQ 核心概念：**
   - NameServer / Broker / Producer / Consumer
   - 消息类型：普通、顺序、延迟、事务消息
   - 事务消息实现原理（半消息 + 回查机制）
   - 消息重试 & 死信队列
   - 与 Kafka 的对比和选型

### 四、数据库 & 缓存（你已有基础，强化即可）
1. MySQL 索引原理（B+树、聚簇/非聚簇、覆盖索引、索引下推）
2. MySQL 事务隔离级别 & MVCC
3. Redis 数据结构、持久化、主从、哨兵、Cluster
4. 缓存穿透/击穿/雪崩 & 解决方案
5. Redis 分布式锁（你项目中用过，要能讲清 Redisson 实现原理）

### 五、分布式系统基础
1. CAP 理论 & BASE 理论
2. 分布式 ID 生成（雪花算法）
3. 分布式事务（2PC、TCC、最终一致性）
4. 一致性哈希

### 六、易宝支付业务相关
- 易宝是第三方支付公司，了解支付流程：下单 → 调用支付网关 → 银行/渠道 → 回调通知
- 支付系统关键点：幂等性、对账、退款、资金安全
- 可能涉及：高并发支付、防重复支付、订单状态机

## 七、Mini Pay 模拟项目真实化增强（项目加分项）

目标：把“技术演示项目”升级为“更像真实支付系统的项目”，面试时可重点讲业务闭环与工程能力。

### P0（第一阶段，优先完成）
1. 支付回调闭环（最关键）
   - 增加“第三方支付平台异步回调”接口
   - 增加签名验签、防重放（nonce + timestamp）、回调幂等
   - 订单状态由回调驱动，和当前同步支付形成双通道校验
   - 验收标准：重复回调 10 次，订单状态只更新一次且链路可追踪

2. 订单状态机完善
   - 增加标准状态：`PENDING`、`PAYING`、`PAID`、`PAY_FAILED`、`CLOSED`、`REFUNDING`、`REFUNDED`
   - 增加状态迁移校验（非法迁移直接拒绝）
   - 验收标准：接口测试覆盖所有合法/非法状态流转

3. 退款流程（全额 + 部分）
   - 新增退款申请、退款回调、退款记录
   - 成功退款后库存回滚，失败可重试
   - 验收标准：部分退款与全额退款都能正确落库并可对账

### P1（第二阶段，体现支付中台能力）
4. 对账中心（T+1）
   - 模拟渠道账单文件导入（CSV 即可）
   - 实现本地订单/支付流水与渠道账单对账
   - 产出差异单（金额不一致、状态不一致、漏单）
   - 验收标准：能生成差异报告并支持“标记已处理”

5. MQ 可靠性增强
   - 增加 Outbox 本地消息表（事务内落库，异步投递）
   - 消费者幂等表（防重复消费）
   - 重试队列 + 死信队列 + 告警日志
   - 验收标准：手动制造消费失败，能自动重试并最终一致

6. 网关限流与风控规则
   - 按用户/IP 做限流（令牌桶或滑动窗口）
   - 增加高频支付拦截与金额阈值校验
   - 验收标准：压测触发后返回明确限流码并记录风控日志

### P2（第三阶段，工程化与可运维）
7. 可观测性增强（监控 + 告警）
   - 接入 Micrometer + Prometheus + Grafana（本地 docker 即可）
   - 增加核心指标：下单 QPS、支付成功率、回调延迟、MQ 积压
   - 验收标准：面试演示可展示实时仪表盘

8. 灰度与配置管理
   - 把关键参数（限流阈值、重试次数、超时）抽到配置中心
   - 支持运行期调参并记录变更
   - 验收标准：不重启服务可动态生效

9. 压测与容量基线
   - 固定压测场景（并发 1/10/30/50）
   - 输出报告：QPS、P95、错误率、瓶颈定位
   - 验收标准：形成可复用压测脚本 + 图表结果

## P0 开发任务拆解（按模块：order/pay/gateway/common/web）

目标：先把“支付回调闭环 + 状态机 + 退款”做完整，形成可演示、可压测、可面试复述的版本。

### 一、`mini-pay-common`（先改，提供统一模型）
1. 状态与常量统一
   - 新增 `OrderStatus` 常量/枚举：`PENDING`、`PAYING`、`PAID`、`PAY_FAILED`、`CLOSED`、`REFUNDING`、`REFUNDED`
   - 新增 `PayStatus`、`RefundStatus` 常量/枚举
   - 输出：所有服务不再写魔法数字

2. DTO 与签名工具
   - 新增 DTO：`PayCallbackRequest`、`RefundRequest`、`RefundCallbackRequest`
   - 新增工具：`SignUtil`（HmacSHA256），含 `sign()` 与 `verify()`
   - 输出：回调接口具备验签能力

3. 数据库脚本补充（`sql/init.sql` 追加增量）
   - `t_pay_record` 增加：`out_trade_no`、`channel_txn_no`、`callback_time`、`callback_raw`
   - 新增 `t_refund_record`
   - 新增幂等表 `t_idempotent_record`（或先用 Redis + 唯一索引二选一）
   - 输出：可追踪回调原文、退款全链路可落库

验收：
1. 所有模块编译通过
2. 数据库可一键初始化并包含新增字段/表

### 二、`mini-pay-gateway`（为回调放行 + 链路追踪）
1. 回调路由放行
   - 在 `AuthGlobalFilter` 白名单新增：`/api/pay/callback/**`、`/api/pay/refund/callback/**`
   - 回调接口不依赖用户 JWT

2. 请求追踪头
   - 统一透传/生成 `X-Request-Id`
   - 写入网关日志，便于联调排错

验收：
1. 回调接口无 token 可访问，业务接口仍需 token
2. 每条请求日志可看到 `X-Request-Id`

### 三、`mini-pay-order`（状态机核心）
1. 状态机校验器
   - 新增合法迁移规则（如 `PENDING -> PAYING -> PAID`）
   - 非法迁移直接拒绝并写链路日志

2. 内部状态更新接口（供 `pay-service` 调用）
   - `markPaying(orderId)`
   - `markPaid(orderId, channelTxnNo)`
   - `markPayFailed(orderId, reason)`
   - `markRefunding(orderId)`
   - `markRefunded(orderId, refundId)`

3. 退款后库存回滚
   - 退款成功后调用 `product-service` 恢复库存
   - 防重复恢复（幂等）

验收：
1. 覆盖合法/非法状态迁移测试
2. 重复回调不会重复改状态/重复回滚库存

### 四、`mini-pay-pay`（回调闭环与退款主逻辑）
1. 支付创建增强
   - 创建支付单时生成 `out_trade_no`
   - 支付单状态改为 `PROCESSING`
   - 调用 `order-service markPaying`

2. 支付回调接口
   - 新增：`POST /api/pay/callback/mock`
   - 验签 + 幂等（同 `out_trade_no + channelTxnNo` 仅处理一次）
   - 成功后：更新支付单、调用 `order-service markPaid`、写链路
   - 失败后：调用 `order-service markPayFailed`

3. 退款接口与退款回调
   - 新增：`POST /api/pay/refund`
   - 新增：`POST /api/pay/refund/callback/mock`
   - 退款回调成功后：更新退款记录 + 调用 `order-service markRefunded`

4. MQ 保持“异步通知”角色
   - 支付/退款结果继续发 RocketMQ（主流程成功后异步发）
   - MQ 失败不影响主事务，保留降级与补偿日志

验收：
1. 支付回调成功/失败路径都能闭环
2. 退款全额/部分都能落库并更新订单状态
3. 重复回调 10 次结果一致

### 五、`mini-pay-web`（演示与联调效率）
1. 测试工具页新增 P0 功能入口
   - 支付回调模拟（成功/失败）
   - 退款申请与退款回调模拟
   - 按订单查看“支付回调/退款回调”链路节点

2. 状态可视化
   - 订单列表显示状态机流转（带时间）
   - 异常原因直接展示（验签失败、重复回调、非法状态迁移）

验收：
1. 前端可完整演示：下单 -> 支付回调 -> 退款 -> 退款回调
2. 每一步都能在链路弹窗看到
